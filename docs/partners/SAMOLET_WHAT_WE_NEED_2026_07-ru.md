# Что конкретно нужно от Самолёта (Задача 07)

Документ для совместного kickoff с техзаказчиком / проектным офисом Самолёта.  
Опирается на: готовность AeroBIM ([`TECHLAB_TASK_07_READINESS_2026.md`](TECHLAB_TASK_07_READINESS_2026.md)), матрицу R1–R15 ([`../samolet-techlab-alignment-2026.md`](../samolet-techlab-alignment-2026.md)), TZ-пакет ([`../tz/README.md`](../tz/README.md)), мировые и российские практики июля 2026 (см. §8).

**Принцип:** AeroBIM уже умеет проверять согласованный комплект. Самолёту нужно поставить **границы истины** (что считать ошибкой), **корпус** (на чём мерить) и **людей** (кто подтверждает TP/FP). Без этого нельзя честно закрыть SLA ≤30 мин и точность на пилоте.

---

## 0. Краткий ответ руководству (1 экран)

| Нужно от Самолёта | Зачем | Блокер без этого |
|-------------------|-------|------------------|
| **1 согласованный комплект** ПД/РД + IFC + ТЗ + расчёты + 2D | Измерить SLA и качество | Нет пилотного KPI |
| **Согласованный набор правил** (не «все СП») | IDS / rule pack = machine-readable EIR | Нет acceptance criteria |
| **≥20 типовых ошибок** с примерами | Калибровка замечаний и приоритетов | Каталог остаётся scaffold |
| **2 инженера-адjudicators** + baseline часов | TP/FP и экономия времени | Нельзя заявить точность / 20% экономии |
| **CDE + версия BCF** | Круг замечаний до стройки | Нет handoff как у Solibri/BIMcollab |
| **Письменный scope memo** | Дисциплины, стадии, что *не* проверяем | Scope creep и ложные ожидания по CV/ГОСТ |

Бюджет TechLab **2 000 000 ₽** должен покрывать именно эту калибровку на корпусе Самолёта, а не «дописать нейросеть с нуля».

## 1. Почему так (мировые практики)

### 1.1 ISO 19650 / EIR → machine-readable checks

По UK BIM Framework и практике ISO 19650 заказчик (appointing party) задаёт **Exchange Information Requirements** и **acceptance criteria**; для структурированных данных критерии превращают в **автоматические правила** (IDS), а не оставляют в PDF-ТЗ.

Следствие для Самолёта: недостаточно «пришлите модели». Нужны **явные критерии приёмки** на пилотный обмен (какие свойства, какие значения, какая стадия). AeroBIM уже проверяет IDS 1.0 + IFC; Самолёту нужно **утвердить содержимое IDS / rule pack** под свой типовой объект.

### 1.2 openBIM loop: IDS → IFC check → BCF

buildingSMART (IDS 1.0 final, июнь 2024; feedback на 1.1/2.0) и кейсы 2025–2026 (в т.ч. Automation in Construction 2026 по IDS+bSDD+BCF) описывают цикл:

1. заказчик фиксирует требования в IDS;  
2. модель проверяется против IDS;  
3. отклонения уходят в **BCF** в CDE / issue hub.

Следствие: Самолёту нужно назвать **CDE / issue tool** и версию BCF (2.1 по умолчанию у AeroBIM), плюс один пилотный проект, куда импортируют BCF.

### 1.3 IDS ≠ геометрия (W78 2024 и industry)

CIB W78 2024 и industry notes (ACCA/Solibri): IDS закрывает **алфавитно-цифровые** требования; clash / clearances / топология — **отдельный** контур.

Следствие: в scope memo Самолёта явно разделить:

- **A)** IDS/атрибуты/ТЗ/расчёты (ядро AeroBIM);  
- **B)** геометрические коллизии (IfcClash, opt-in);  
- **C)** «умный CV по сканам» — **не** критерий приёмки MVP (advisory).

### 1.4 KPI координации 2025–2026: не «число clash»

Практика VDC 2026 (и оценки FP до ~60% у «сырого» clash): считают **decision velocity**, долю подтверждённых findings, готовность модели к стройке — не сырой счётчик пересечений.

Следствие: от Самолёта нужны **люди на adjudication** и согласие на interim KPI **TP ≥ 60%**, а не немедленные «>90%» без разметки (90% — цель после labeled corpus, см. [`../pilot-claim-boundary-2026.md`](../pilot-claim-boundary-2026.md)).

## 2. Почему так (российские практики)

### 2.1 Нормативный контур ТИМ

- ГрК ст. 57.5, ПП № 331 (случаи обязательной ИМ), ПП № 614 (состав/форматы ИМ);  
- СП 333.1325800.2020 (формирование ИМ по ЖЦ);  
- ГОСТ Р ИСО 19650-1/2-2020;  
- с **01.03.2026** — усиление требований к составу ЦИМ, дополняющей графическую часть ПД (приказ Минстроя / проект состава ЦИМ).

Практика 2025–2026 (экспертиза ЦИМ): типичные провалы — **нет требований к ИМ в ТЗ**, ошибки IFC-экспорта, расхождение ЦИМ с текстовой/графической ПД, «отчёт по коллизиям не бьётся с моделью».

Следствие для Самолёта: пилотный комплект должен включать **ТЗ/EIR-фрагмент с требованиями к ИМ** (хотя бы для одного раздела) и **IFC, реально выгруженный** из рабочей цепочки Самолёта, а не «красивый демо-IFC».

### 2.2 Что реально сдают / проверяют в РФ

Типовой набор 2025: IFC4 (LOD по стадии), PDF из модели, спецификации, отчёт коллизий, сверка объёмов/ТЭП. Для РД критична **взаимоувязка с утверждённой ПД** (габариты, нагрузки, материалы).

Следствие: в пилоте Самолёта желательна **пара ПД↔РД** (хотя бы один раздел: АР или КЖ + смежный), иначе нельзя показать кросс-документный кейс Task 07.

### 2.3 Корпоративный EIR / BEP

Российские гайды 2026 по ТЗ на BIM: заказчик добавляет к обычному ТЗ блоки LOD, форматы IFC, классификатор, clash-матрицу, СОД (CDE), BEP.

Следствие: Самолёту не обязательно отдавать весь корпоративный EIR. Нужен **выписка на пилот** (1–2 страницы + IDS/таблица свойств).

## 3. Блок A — данные (Week 1, blocking)

Один **типовой жилой** (или иной согласованный) объект / секция. Объём: чтобы анализ укладывался в **≤30 мин** на согласованном железе (не весь портфель).

### A1. BIM

| Артефакт | Требование | Формат |
|----------|------------|--------|
| Федеративная или дисциплинарная модель | Стадия П или Р (зафиксировать) | **IFC 2x3 / 4 / 4x3** (указать схему) |
| Нативная модель (опционально) | Для разбора экспорта | RVT/… (локально, NDA) |
| Отчёт/матрица коллизий (если есть) | Для калибровки FP | PDF/Excel/BCF |

### A2. Документация

| Артефакт | Требование |
|----------|------------|
| Фрагмент **ПД** | PDF (+ XML ПЗ если есть) по выбранным разделам |
| Фрагмент **РД** | PDF/чертежи тех же разделов |
| **ТЗ** на проектирование / выписка EIR | RU; свойства и критерии приёмки |
| **Расчёты** | Тексты/таблицы нагрузок, площадей, огнестойкости — то, что сверяют вручную |
| 2D | PDF (вектор предпочтительнее скана); DWG — если есть, отдельно пометить |

### A3. Машиночитаемые требования

| Артефакт | Требование |
|----------|------------|
| **IDS 1.0** *или* таблица свойств → мы соберём IDS | Мин. 15–30 правил на пилотные дисциплины |
| Корпоративный классификатор / маппинг к IFC | Хотя бы для FireRating, площади, толщин, классов бетона |

Пути кладутся в локальный манифест по шаблону  
`samples/benchmarks/project-package-samolet-pilot-v1.template.json` (gitignored customer pack).

## 4. Блок B — правила и границы (Week 1, blocking)

### B1. Scope memo (1–2 стр., подпись техзаказчика)

Обязательные пункты:

1. Дисциплины пилота (например: АР + КЖ + ОВ).  
2. Стадия (П / Р).  
3. Что **входит** в автопроверку: IDS-атрибуты, кросс-док ТЗ↔расчёт↔IFC, drawing annotations, optional clash.  
4. Что **не входит**: полный СП/ГОСТ, замена ГГЭ, «CV понимает любой скан как человек», авто-sign-off.  
5. Политика коллизий: geometric clash влияет на `passed` или только advisory (`AEROBIM_CLASH_AFFECTS_PASS`).  
6. Язык замечаний: RU (default) / EN.

### B2. Каталог типовых ошибок (≥20)

Для каждой записи:

- `error_id`, дисциплина, title_ru;  
- пример «как выглядит в документах Самолёта»;  
- ожидаемый класс: IDS / CROSS_DOC / SPATIAL / DRAWING;  
- severity hint;  
- ссылка на файл в комплекте A.

Шаблон: [`../../samples/benchmarks/samolet-typical-errors-catalog.json`](../../samples/benchmarks/samolet-typical-errors-catalog.json).

Без этого пилот калибрует «вслепую» и R14 остаётся scaffold.

## 5. Блок C — люди и процесс (Week 1–2, blocking для KPI)

| Роль | От Самолёта | Обязанность |
|------|-------------|-------------|
| Sponsor / техзаказчик | 1 контакт (уровень Горелик / Хасанов) | Утверждение scope memo |
| BIM/координатор | 1 | Выгрузка IFC, CDE, BCF import |
| Инженер-эксперт (adjudicator) | **≥2** | Разметка TP/FP по замечаниям AeroBIM |
| Нормоконтроль / ГИП (опц.) | 1 | Согласование rule pack |

**Baseline:** часы ручной проверки **того же** комплекта (до или параллельно пилоту) — иначе KPI «−20% времени» не считается ([`../pilot-claim-boundary-2026.md`](../pilot-claim-boundary-2026.md)).

**Adjudication:** журнал по [`../samolet-kpi-adjudication-template-2026.md`](../samolet-kpi-adjudication-template-2026.md); целевой interim **TP/(TP+FP) ≥ 60%**.

## 7. Блок E — юридическое и орг. (до старта)

1. NDA / акт передачи пилотных данных.  
2. Согласие: пилотные метрики публикуются в TechLab **без** раскрытия коммерческой тайны объекта (агрегированно).  
3. Контакт ИБ: куда класть данные (не в публичный git).  
4. Подтверждение бюджета пилотного тестирования TechLab (**2 000 000 ₽**) и владельца со стороны Самолёта.

## 8. Что Самолёту **не** нужно приносить (уже есть у AeroBIM)

- Движок IFC/IDS/cross-doc/clash, API, browser review, upload, BCF export, HITL remarks, SLA-rail, claim boundary.  
- «Обучающую выборку на миллион чертежей» для MVP sign-off — детерминированный контур.  
- Полный свод СП как XML — на пилоте только **согласованный** rule pack.

## 11. Источники (июль 2026)

**Мир / openBIM**

- buildingSMART IDS 1.0 (final 2024-06); IDS+bSDD+BCF workflows (Buildings 2025; Automation in Construction 2026, doi:10.1016/j.autcon.2026.107043).  
- UK BIM Framework / ACCA BibLus — EIR задаёт acceptance criteria; IDS делает их machine-checkable на IFC.  
- ISO 19650 **DIS 2026** (март 2026): сдвиг BIM→IM; в черновике EIR→IPR, BEP→IPP — для пилота Самолёта достаточно текущей терминологии EIR + IDS; не ждать финальной редакции.  
- CIB W78 2024 — Automated compliance checking using IDS (limits: alphanumeric, not geometry).  
- Industry 2025–2026: clash FP noise; KPI shift to confirmed findings / decision velocity (VDC practice notes).  
- Solibri / BIMcollab: clash + rule check + BCF as coordination loop (complement, not replace).

**Россия**

- ГрК 57.5 / 57.7; ПП 331; ПП 614 (состав/форматы ИМ); СП 333.1325800.2020; ГОСТ Р ИСО 19650-1/2-2020.  
- С 01.09.2024 — ЦИМ в экспертизу для ряда договоров; с **01.03.2026** — ужесточение состава ЦИМ / реестра требований в предмете экспертизы.  
- Практика экспертизы ЦИМ 2025–2026: нет требований к ИМ в ТЗ, ошибки IFC-экспорта, расхождение ЦИМ↔ПД, clash-отчёт не бьётся с моделью.  
- Гайды 2026: в ТЗ на BIM — LOD/LOIN, IFC, классификатор (КСИ), clash-матрица, СОД.
